## Build, Lint, and Test Commands

### Core commands

- Build/install CLI locally: `go install .`
- Run the app from source: `go run ./main.go`
- Run all tests: `go test ./...`
- Run verbose race-enabled suite: `go test -v -race ./...`
- Run linter: `golangci-lint run ./...`
- Apply standard Go fixes: `go fix ./...`
- Regenerate mocks/code: `go generate ./...`

### Single-test commands

- Run one named test in one package:
  - `go test ./internal/application -run '^TestRunReviewsOnlyNewDiffs$'`
- Run one named test with verbose output:
  - `go test -v ./internal/adapters/github -run '^TestClientGetMergeRequestChanges$'`
- Run multiple tests by regex:
  - `go test ./internal/adapters/gitlab -run 'TestClient(AddMergeRequestDiscussion|GetMergeRequestChanges)'`
- Run a package without cache while debugging:
  - `go test -count=1 ./internal/adapters/config -run '^TestNewConfigWithDefaults$'`
- Run all tests in one package:
  - `go test ./internal/adapters/ai`

### Focused iteration workflow

1. Edit code.
2. Run targeted tests for the touched package.
3. Run `golangci-lint run ./...` if Go files changed.
4. Run `go generate ./...` if interfaces or generated mocks may be affected.
5. Before finishing, run `go test -v -race ./...`.

## Git Hooks and Generated Code

- Pre-commit hook in `.lefthook.yml` runs:
  - `go generate ./...`
  - `golangci-lint run`
  - `go fix ./...` on staged Go files
- Pre-push hook runs:
  - `go test -v -race ./...`
- Generated mocks live in `internal/domain/mocks/*.gen.go`.
- Mocks are generated by `mockery` via `//go:generate go tool mockery` in `internal/domain/use_cases.go`.
- Do not hand-edit generated `*.gen.go` files unless you are deliberately replacing generated output.

## Formatting and Imports

- Use standard Go formatting; the repo enables both `gofmt` and `goimports` in `.golangci.yml`.
- Keep imports grouped the normal Go way:
  1. standard library
  2. module-local and third-party imports
- Prefer import aliases only when needed to disambiguate package names or improve readability, e.g. adapter aliases in `main.go`.
- Keep files ASCII unless the file already requires something else.
- Avoid comment noise; add comments only when the code is not obvious from names and structure.

## Code Style Expectations

- Follow existing Go idioms, not custom patterns.
- Keep functions small enough to satisfy enabled linters such as `cyclop` and `funlen`.
- Prefer explicit constructor functions like `NewClient`, `NewReviewer`, `NewAIProvider`.
- Keep package APIs narrow and cohesive.
- Prefer concrete structs behind domain interfaces/ports.
- Prefer direct field access on concrete config structs over getter wrappers.
- Return concrete values plus `error`; do not panic in normal flow.
- Use zero values and defaults intentionally.
- Prefer `strings.Builder`, `maps`, `slices`, and other stdlib helpers already used in the codebase when they simplify logic.

## Naming Conventions

- Packages use short lowercase names: `ai`, `github`, `gitlab`, `config`, `domain`, `application`.
- Exported types and functions use Go PascalCase.
- Unexported helpers use camelCase.
- Interface names in `internal/domain/use_cases.go` end with `Port`.
- Test helper constants often use `err...` and `test...` prefixes; keep that convention in tests.
- Prefer names that describe role in the workflow, e.g. `mrProvider`, `aiProvider`, `commentPrefix`, `reviewText`.
- Keep environment-backed config fields aligned with env variable names where practical.

## Error Handling

- Wrap returned errors with context using `fmt.Errorf("...: %w", err)`.
- Use short, action-oriented messages like `create config`, `review code`, `parse CI_MERGE_REQUEST_IID`.
- Log recoverable problems with `logger.Warn(...)` and continue when current behavior does so.
- Return fatal workflow failures to callers instead of swallowing them.
- Close HTTP response bodies with deferred cleanup.
- If a linter suppression is necessary, keep it tight and local, e.g. a line-level `//nolint:gosec` with clear intent.

## Types and Architecture

- Keep domain contracts in `internal/domain`.
- Keep shared runtime/config data as concrete domain structs when mocking would only add indirection.
- Put orchestration logic in `internal/application`.
- Put external service implementations in `internal/adapters/...`.
- New VCS or AI providers should usually be added as new adapters implementing an existing domain port.
- Thread `context.Context` through external calls and long-running operations.
- Read env vars only in `internal/adapters/config`; pass typed config values downward from there.
- Prefer dependency injection via Fx wiring in `main.go` rather than package globals.

## Testing Guidelines

- Testing library: standard `testing` package with `testify/mock` generated by `mockery`.
- Most tests are package-local unit tests in `*_test.go` files.
- Many tests call `t.Parallel()` early; preserve that where safe.
- Use `t.Setenv(...)` for configuration-driven tests.
- For HTTP client tests, prefer `internal/testutil/httpstub` over real network calls.
- For domain port mocks, use generated mocks in `internal/domain/mocks`; for config, prefer concrete fixtures unless a remaining interface truly needs a mock.
- Keep assertions explicit and readable; existing tests often use `t.Fatalf` / `t.Fatal` directly.

## Linting Rules That Matter in Practice

- `errcheck`: check returned errors.
- `wrapcheck`: wrap errors crossing boundaries.
- `wsl_v5`: keep whitespace readable; add blank lines where the linter expects them.
- `gosec`: be careful with dynamic URLs, tokens, and HTTP requests.
- `noctx`: use context-aware APIs when available.
- `thelper`: mark real test helpers with `t.Helper()`.

## Agent Do/Don't Notes

- Do read existing patterns in neighboring files before changing structure.
- Do update tests when behavior changes.
- Do regenerate mocks if a domain interface changes.
- Do not introduce a new framework or config system without a strong reason.
- Do not bypass hooks with `--no-verify` unless explicitly told.
- Do not commit secrets or real tokens into docs, code, or tests.
- Do not replace simple stdlib solutions with extra dependencies.
